# Dev Cadence 的软件开发实施理念

## 软件开发实施的目标不是写代码

软件开发实施的目标，是把已经确认的工作项责任可靠地转化为可以运行、可以验证、可以维护的软件行为。

代码是实施的重要产物，但代码数量、提交数量和开发速度都不能单独说明实施产生了价值。一项实现即使能够编译、局部测试通过，也可能误解需求、偏离设计、遗漏异常路径、破坏既有行为，或者无法证明实际交付的版本就是被评审和验证的版本。

Dev Cadence 不以“代码已经生成”判断实施是否完成。它关注的是：工作项理解是否被正确落实，技术判断是否有依据，变更是否受到控制，预期行为是否有证据支持，风险是否被看见，以及形成的软件是否足以承担确认的工作项责任。

## 实施是对工作项责任的技术兑现

单卡需求分析回答当前工作项为什么存在、必须提供什么结果以及如何判断完成。软件开发实施则回答系统如何可靠地产生这些结果。

两者之间不是从需求条目到代码文件的机械映射，而是一条责任转换链：

```text
用户和业务需要
-> 工作项责任
-> 可交付的预期行为
-> 系统责任和技术边界
-> 可执行的实施步骤
-> 经过验证的软件行为
```

实施必须能够解释每项重要技术工作如何服务于已经确认的预期行为。无法建立这种关系的代码，可能是范围外工作、未经确认的需求决策，或者尚未证明必要的技术复杂度。

实施也不能把技术限制静默改写成需求规则。当系统无法按原有理解实现需求时，团队需要暴露冲突、重新确认责任或调整方案，而不是让代码中的偶然行为成为事实上的需求定义。

## 从确认的理解开始，而不是从原始指令开始

用户提出的功能名称、修改建议或修复要求，是实施工作的输入，但不一定已经具备直接编码的条件。

实施开始前至少需要明确：

- 当前要实现或恢复什么行为；
- 哪些用户、场景和结果属于本次范围；
- 哪些内容明确不在本次范围；
- 什么条件能够证明结果成立；
- 当前方案依赖哪些事实、接口、架构和环境；
- 哪些问题仍然未知，是否会阻止安全实施。

这些内容不要求所有不确定性都被消除，但必须足以界定当前变更的责任。缺少关键理解时，立即编码不会减少不确定性，只会把不确定性藏进实现。

如果实施期间发现需求本身不清楚、方案与架构冲突或依赖并不存在，正确动作不是继续猜测，而是回到能够解决该问题的最早环节。

## 设计、计划、编码、测试和评审共同构成实施

软件开发实施不是单独的编码阶段。技术设计、实施计划、代码变更、测试验证、调试和 Code Review 共同完成从工作项责任到可信软件的转换。

```text
确认的预期行为
-> 技术设计
-> 实施计划
-> 代码与测试
-> 持续检查和调试
-> Code Review
-> 整体验证
-> 可供业务验收的实现结果
```

这些活动承担不同责任，但不能彼此替代：

- 技术设计决定系统应该承担什么责任以及责任如何分布；
- 实施计划把设计变成可执行、可检查的变化顺序；
- 代码和配置形成实际的软件行为；
- 测试与验证提供行为符合预期的证据；
- 调试识别现象背后的真实原因；
- Code Review 独立检查实现是否正确、完整并符合约束。

只完成其中一部分，不能形成可信的实施结果。设计文档不能代替运行行为，测试数量不能代替正确范围，Review 结论不能代替验证证据，代码提交也不能代替实施完成。

## 技术设计是系统责任的判断

技术设计不是把需求换成模块、接口和数据表，也不是为已经决定的代码结构补充说明。

可靠的技术设计需要判断：

- 系统必须提供哪些可观察行为；
- 哪些组件分别承担什么责任；
- 责任之间通过什么稳定边界协作；
- 数据、状态和错误如何流动；
- 既有契约和兼容性如何保持；
- 安全、性能、可访问性和运维约束如何影响方案；
- 失败、恢复和边界情况由谁处理；
- 哪些决策是当前必需的，哪些复杂度可以推迟。

设计的价值不在于提前描述每一行代码，而在于让重要责任、约束和取舍在编码前可以被检查。设计应该为实施提供方向，同时保留实现细节在证据出现后调整的空间。

现有代码是重要事实，但不是天然正确的设计答案。当现有结构与确认的工作项责任冲突时，实施需要明确处理这种冲突，不能因为某种结构已经存在就默认继续复制。

## 实施计划是对变化顺序的推理

实施计划不是任务标题的清单，也不是对技术方案的重复描述。它需要把方案拆成能够逐步建立、逐步验证并在失败时定位原因的工作单元。

一个可靠的计划应该说明：

- 每一步准备改变什么行为或结构；
- 为什么这一步现在必要；
- 将修改哪些明确的技术边界；
- 用什么测试或检查证明这一步成立；
- 预期先看到什么失败，再看到什么通过；
- 这一步完成后，哪些风险已经降低，哪些仍然存在。

计划的粒度应该服务于理解和反馈。过大的步骤把多个假设混在一起，使失败难以定位；过小而没有独立意义的步骤只会增加操作和记录负担。

计划也不是永久有效的承诺。进入编码前和实施过程中，如果需求、设计、依赖或代码基线发生了实质变化，必须重新判断计划是否仍然成立。继续执行已经失效的计划，不是遵守纪律，而是在有序地实施错误结论。

## 实现必须忠于已确认的行为边界

实施者会在编码过程中遇到大量局部选择，但不能因此获得静默改变工作项责任的权限。

实现可以在确认的技术边界内选择更清晰、更简单或更可靠的方式，但以下变化不能作为普通编码细节处理：

- 改变用户能够获得的结果；
- 增加或删除业务规则；
- 改变公共接口或外部数据契约；
- 缩小已经确认的场景或异常覆盖；
- 用技术便利替代已经确认的安全或一致性要求；
- 引入超出当前目标的新预期行为。

遇到这些情况时，需要显式说明变化及其影响，并返回相应的需求或设计判断。实现不能因为修改代码更方便，就自动把“当前能做什么”变成“系统应该提供什么”。

同样，实施不应为了看起来完整而扩展范围。未来可能有用的抽象、未经要求的功能和没有当前证据支持的通用化都会扩大验证面，并降低当前结果的可信度。

## 小步实施服务于快速获得真实反馈

小步实施的价值不在于产生更多提交，而在于缩短假设与证据之间的距离。

每个有意义的实施步骤都应尽量形成一条短反馈链：

```text
一个明确预期
-> 一项受控变化
-> 一次针对性检查
-> 一个可以解释的结果
```

当变化足够聚焦时，失败更容易归因，Review 更容易理解，回退和修正也更安全。多个相互独立的目标被混入一次大改动时，即使最终测试失败，团队也难以判断是哪项假设、哪处实现或哪条测试出了问题。

小步不等于局部主义。每一步可以很小，但所有步骤最终必须组合成完整的预期行为。只优化局部实现而没有验证端到端结果，仍然可能得到无法完成工作项目标的软件。

## 测试与验证属于实施本身

测试不是代码完成后的质量检查，也不是专门由下游角色补充的证明材料。测试与验证参与定义实现边界、推动设计反馈，并证明实际行为是否符合预期，因此属于软件开发实施本身。

测试先行的核心价值，是在实现之前把预期行为变成可执行判断：

```text
明确预期
-> 观察预期失败
-> 编写最小实现
-> 观察预期通过
-> 在行为保护下改善结构
```

失败证据说明测试能够观察到缺失或错误行为；通过证据说明当前实现满足了该判断。只有通过而没有证明测试对目标行为敏感，可能只是测试从未真正覆盖风险。

测试驱动不是为了制造形式上的 RED，也不是要求所有文件变化都新增测试。已经存在且受到有效测试保护的行为保持型重构，可以在绿色基线上小步调整；保护既有行为的新 characterization 或 contract test，则需要通过可逆扰动或等价方法证明测试确实能够检测变化。

文档、格式或其他不改变可执行行为的工作，不应为了满足仪式而编造失败测试。测试形式必须与风险和行为相匹配。

## 功能开发、Bug 修复和重构具有不同的实施责任

三类开发工作都需要受控实施和验证，但它们要证明的事情不同。

功能开发建立新的预期行为，核心问题是：确认的目标能力是否被完整实现，正常、替代和失败路径是否产生预期结果。

Bug 修复恢复已经存在的预期行为，核心问题是：原始问题能否可靠复现，根因是否得到证据支持，修复是否作用于根因，以及受影响范围是否没有产生回归。

重构改善内部结构而不主动改变预期行为，核心问题是：行为基线是否可信，结构目标是否真正达成，以及外部契约和可观察结果是否保持兼容。

```text
功能开发：建立新的正确行为
Bug 修复：恢复被破坏的正确行为
重构：保持正确行为并改善内部结构
```

不能用同一种证据笼统证明三类工作。功能开发不能只证明没有回归，Bug 修复不能只证明代码发生变化，重构也不能用新增行为掩盖行为漂移。

## 调试的目标是建立因果解释

错误信息、失败测试和异常现象只是调查入口，不等于根因。

可靠调试需要从可复现现象开始，收集能够区分不同假设的证据，沿着数据、状态和调用关系定位原因，再用修复后的结果验证因果判断。没有根因证据时反复修改代码，可能暂时改变症状，却会增加新的变量并掩盖真实问题。

调试应该能够回答：

- 观察到了什么稳定现象；
- 哪些条件会触发或消除现象；
- 数据或控制流在哪里偏离预期；
- 什么证据支持当前根因；
- 修复为什么能够消除这个原因；
- 什么回归检查可以防止同类问题重新出现。

AI 尤其容易根据熟悉的错误模式快速提出修复。模式识别可以生成调查假设，但不能替代当前代码和运行环境中的因果证据。

## 失败是对当前理解的检验

实施中的失败不都属于代码错误。测试可能有缺陷，运行环境可能不可用，需求可能含糊，设计可能与架构冲突，依赖也可能缺失。

如果不区分失败性质，团队就会在错误层级反复修改：用放宽测试掩盖实现错误，用代码补丁掩盖需求冲突，用重复执行掩盖环境问题。

可靠实施需要根据证据判断失败最早由哪项错误理解产生：

```text
需求理解错误 -> 回到需求判断
设计假设错误 -> 回到技术设计
计划或依赖错误 -> 调整实施计划
实现错误 -> 修正代码和测试
测试错误 -> 修正测试资产
环境问题 -> 明确阻塞条件
```

返回较早环节不是流程倒退，而是保护后续工作不再建立在失效结论上。历史记录可以保留，但已经失效的确认、测试和 Review 证据不能继续代表当前实现。

对同一失败的再次尝试必须基于新的证据、纠正动作或环境变化。没有任何条件变化的重复执行，只是在重复观察已知结果。

## Code Review 是实施的一部分

Code Review 的目标不是确认代码风格，也不是为提交增加批准标记。它提供独立视角，检查实际变化是否忠于需求和设计、是否引入缺陷、是否遵守适用规则，以及测试是否真正覆盖重要风险。

高价值 Review 关注：

- 实现与确认目标、范围和验收条件是否一致；
- 实际代码路径是否产生预期行为；
- 错误处理、边界条件和兼容性是否可靠；
- 安全、性能、可访问性和运维风险是否得到适当处理；
- 测试是否能够检测重要错误，而不只是执行了代码；
- 实现是否引入不必要复杂度、重复责任或范围外变化。

Review 结论必须针对明确的变更身份。只审查工作区的某个临时状态，却提交了不同内容，不能证明提交已经被评审。实施需要能够说明被评审、被测试和被记录的代码之间是什么关系。

发现问题后，修复本身也是新的实施变化，需要重新接受相应检查。不能因为问题来自 Review，就默认修复正确或不需要再次验证。

## 证据必须对应实际实现

可信实施不仅需要证据，还需要知道证据证明的是哪个对象。

需求、技术方案、实施计划、代码版本、测试结果和 Review 结论之间必须保持可追溯关系：

```text
确认的需求和设计
-> 本次实施范围
-> 实际代码变化
-> 针对该变化执行的测试
-> 针对该变化完成的 Review
-> 当前实现结论
```

旧版本上的测试通过不能自动证明新版本正确，未执行的测试不能作为通过证据，其他分支上的 Review 也不能证明当前提交已经审查。

证据的目的不是增加记录数量，而是支持判断和变化。当代码、需求或方案变化时，团队可以沿着关系识别哪些结论仍然有效、哪些检查需要重做、哪些风险需要重新评估。

如果无法确认测试和 Review 对应的实现身份，就必须如实说明证据边界，不能用模糊的“已经验证”代替。

## 完整验证比局部通过更重要

开发过程中的聚焦测试帮助快速反馈，但最终仍需要从确认的实施目标审视完整结果。

可靠的验证需要覆盖：

- 每项重要需求或验收条件；
- 用户可观察的主要行为；
- 与变更有关的替代、失败和恢复路径；
- 受影响的既有行为和公共契约；
- 适用的构建、静态检查、集成检查和运行环境；
- 无法执行的检查及其原因；
- 尚未消除的残余风险。

局部单元测试通过不能证明端到端结果成立，构建成功不能证明行为正确，人工观察也不能自动替代可重复检查。不同证据回答不同问题，必须与风险相匹配。

验证结论只能基于实际执行的证据。必要行为没有得到验证，或者阻塞失败仍然存在时，实施不能以“可能没问题”进入下一步。非阻塞的未覆盖区域和残余风险可以保留，但必须被明确呈现给后续业务判断。

## 风险必须被暴露，而不是被措辞消除

软件实施无法消除所有风险，但必须区分已经证明的结果、尚未覆盖的区域和已知问题。

以下内容不能被笼统的“测试通过”隐藏：

- 被跳过或无法运行的必要检查；
- 没有覆盖的验收条件；
- 依赖外部环境才能确认的行为；
- 已知但暂未处理的兼容性、安全或性能风险；
- Review 中尚未解决的重要发现；
- 实现与原计划之间的实质偏差。

风险说明不是免责文字。它需要解释风险来自哪里、可能影响什么、为什么当前没有消除，以及后续决策者是否具备接受它所需的信息。

验证没有发现失败，不等于证明不存在风险。相反，明确证据边界可以让团队对当前结果作出更准确的判断。

## 实施记录服务于恢复、审查和变化

软件开发实施需要持久化关键判断和证据，但记录不是实施价值本身。

有效记录应该帮助团队：

- 在跨会话或跨人员协作时恢复当前上下文；
- 知道当前实现依据哪一版需求、设计和计划；
- 识别哪些任务已经完成以及由什么证据支持；
- 理解重要失败、修正和方案变化；
- 确认测试、Review 与实际代码的对应关系；
- 在后续变化发生时判断哪些证据已经失效；
- 向业务验收提供清晰的实现结果和残余风险。

没有决策价值的重复记录会增加同步成本，并使真正重要的证据更难发现。Dev Cadence 追求的是一条完整、可检查的实施证据链，而不是尽可能多的实施文档。

提交保存工程进度，不能代替阶段判断；阶段记录保存实施上下文，也不能代替实际测试；自动生成的总结不能代替原始可检查结果。

## AI 应该帮助实施，而不是代替工程判断

AI 可以快速阅读代码、提出方案、拆解计划、生成实现、编写测试、调查失败和执行 Review，但速度和表达能力不能证明软件正确。

在 Dev Cadence 中，AI 的实施职责是：

- 把确认的预期行为转化为清晰的技术责任和实施步骤；
- 在编码前暴露重要假设、依赖和风险；
- 通过测试先行和小步变化缩短反馈周期；
- 根据当前代码和运行证据调查问题；
- 保持实现、测试、Review 和记录之间的可追溯关系；
- 发现需求、设计或环境冲突时返回正确环节；
- 如实报告失败、跳过的检查、证据边界和残余风险；
- 只在新鲜证据支持时作出完成或就绪判断。

AI 不应该为了完成计划而编造行为、放宽有效测试、隐藏失败或把自己的需求判断写进代码。它也不应该因为生成了大量代码、测试或记录，就声称实施已经可靠完成。

用户确认的目标不能替代技术证据，技术证据也不能替代用户的业务验收。AI 的责任，是让两者之间的关系清楚、诚实并且可以检查。

## 怎样判断软件开发实施已经产生价值

实施是否有效，不由代码是否提交决定，而由团队能否清楚回答以下问题决定：

- 本次实现兑现了什么已经确认的工作项责任；
- 技术设计如何把预期行为转化为系统责任；
- 实际变化是否保持在确认的范围内；
- 每项重要行为由什么测试或检查证明；
- 功能、修复或重构特有的目标是否已经满足；
- 失败是否得到正确分类和处理；
- Code Review 检查的是不是实际提交的变化；
- 聚焦测试和整体验证是否覆盖了重要风险；
- 哪些检查被跳过，哪些区域仍然没有证据；
- 当前还存在哪些残余风险；
- 需求、设计、计划、代码、测试和 Review 是否对应同一实施结果；
- 当前结果是否已经具备进入业务验收的条件。

如果这些问题仍然无法回答，增加更多代码、测试数量或记录篇幅不会让实施变得可信。团队需要回到尚未形成可靠判断的环节继续工作。

## 实施完成不等于软件交付完成

软件开发实施完成，意味着已经形成一个符合确认需求、经过适当测试和 Review、证据边界清楚、残余风险可见的软件结果。

它不自动意味着用户已经接受结果，也不自动意味着代码应该合并、发布或部署。业务验收需要判断软件是否真正解决目标问题；集成、发布和部署还需要各自的环境、权限、风险和回退判断。

```text
单卡需求分析形成实施理解
-> 软件开发实施形成可信的软件结果
-> 业务验收判断结果是否可接受
-> 集成、发布和部署完成后续软件交付
```

区分这些责任，不是把交付割裂成互不相关的手续，而是避免用某一环节的成功替代整个交付结论。

## Dev Cadence 的软件开发实施承诺

Dev Cadence 不以生成更多代码为目标。它希望帮助用户完成一系列可靠的工程判断：

```text
我们知道要兑现什么工作项责任
-> 我们把责任转化成了清晰的技术设计
-> 我们形成了可以逐步验证的实施计划
-> 我们通过小步变化建立了软件行为
-> 我们用测试和调试持续检验当前理解
-> 我们让实际变化接受了独立 Review
-> 我们用与实现版本对应的证据验证了完整结果
-> 我们如实说明了证据边界和残余风险
-> 我们形成了可以进入业务验收的软件
```

代码、测试、提交和记录都是这些判断的载体。

Dev Cadence 真正要保护的，是从已经确认的工作项责任到可信软件行为的转换过程，以及这份责任在设计、计划、编码、测试、Review、验证和后续变化中不被丢失。
