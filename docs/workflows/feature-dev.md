# 功能开发流程

本文说明 Dev Cadence 的 `feature-dev` 业务流程，包括适用场景、阶段目标、主要产出以及与其他 workflow 的边界。本文面向 Dev Cadence 维护者、流程评审者和希望了解交付模型的使用者。

目标仓库中的实际执行规则以 `src/skills/feature-dev/SKILL.md` 生成的安装内容为准。本文用于解释业务流程设计，不替代 workflow skill。

## 目的

功能开发流程用于交付新增的用户可见或系统可见能力，或者有意改变既有预期行为。流程从单个明确工作项开始，将其推进到经过技术设计、实施、系统验证和业务验收的可交付结果。

## 适用场景

- 新增产品能力、平台能力、接口、配置行为或其他可观察功能。
- 有意改变系统当前的预期行为。
- Technical Task 的目标是新增系统能力，而不仅是保持行为的内部整理。
- 实施一个已经规划的 `S-nnn` Story，或者适合通过功能开发交付的 `T-nnn` Technical Task。

## 不适用场景

- 已有预期行为没有正常工作：使用 `bug-fix`。
- 只改善内部结构且不主动改变外部可观察行为：使用 `refactor`。
- 从模糊想法建立或更新 PRD：使用 `discovery`。
- 从 PRD 拆分和维护 Feature、Story、Bug、Technical Task 与 Roadmap：使用 `work-item-planning`。

## 流程阶段

| 阶段 | 目标 | 主要产出 |
|---|---|---|
| 1. 需求确认 | 读取工作项卡片，确认本次交付使用的卡片版本、选定范围、验收标准、明确排除项、风险和待决问题；用户确认本次交付边界。 | 需求确认记录 |
| 2. 技术方案 | 基于已确认需求和当前系统现状，明确实现路径、系统边界、关键取舍、约束、风险以及需要用户决策的事项。 | 技术方案 |
| 3. 实施计划 | 将技术方案拆成可执行、可验证的开发步骤，安排 TDD 小循环、测试资产、联调和接入顺序。 | 实施计划 |
| 4. 开发实施 | 按计划完成代码、配置、文档、脚本、模板或其他项目资产；完成配套测试、检查和代码审查。 | 可工作的交付物；测试资产；实施记录；代码审查报告 |
| 5. 系统测试 | 根据需求、技术方案、计划和实施结果验证整体系统行为；记录执行证据、跳过项、剩余风险和是否可以进入业务验收的结论。 | 系统测试报告 |
| 6. 业务验收 | 用户根据业务目标、验收标准、系统测试结果和剩余风险决定接受、接受风险后接受或拒绝。 | 业务验收记录 |

## 工作项与运行记录

- 每次功能开发必须关联一个 `S-nnn` Story 或 `T-nnn` Technical Task 卡片。
- `feature-dev` 不创建工作项卡片；入口没有找到对应卡片时，必须先进入 `work-item-planning` 的单项登记模式。
- 工作项卡片是长期需求定义；需求确认记录是本次运行使用的卡片版本和交付范围快照。
- 需求确认记录必须引用工作项 ID、卡片路径和卡片版本，不完整复制卡片并形成第二份权威需求。
- workflow 期间发现工作项需要实质升版时，先更新卡片及其 Change Log，再判断当前 run 应返回哪个阶段重新确认。

## 关键规则

- 技术方案和实施计划开始前，需求确认必须得到用户确认。
- 进入开发实施前，重新核对卡片版本、已确认需求、技术方案、实施计划、权威设计资料、依赖和当前代码；只有受影响内容失效时才返回最早受影响阶段，无关变化不阻止实施。
- 实施计划只服务于开发实施，不替代系统测试或业务验收。
- 开发实施包含功能实现及相应的单元测试、集成测试、前端测试、契约测试、mock、检查脚本或检查清单。
- 系统测试关注整体系统是否满足已确认目标，不替代开发阶段应建立的自动化测试资产。
- 阻塞性验证结果不能进入业务验收；返工必须回到最早受影响阶段。
- 可工作的交付物可以是代码、配置、文档、脚本、模板、prompt、规则或其他项目资产。
- 完成业务验收和 Completion 后，应将工作项状态及交付引用回写到卡片和 Roadmap。

## 与其他 Workflow 的关系

```text
discovery -> work-item-planning -> S / T card -> feature-dev
```

用户可以直接提出单项功能请求，但 `using-dev-cadence` 必须先检查卡片。没有卡片时先由 `work-item-planning` 创建和确认卡片，再进入 `feature-dev`。
