# 重构流程

本文说明 Dev Cadence 的 `refactor` 业务流程，包括适用场景、阶段目标、主要产出以及与其他 workflow 的边界。本文面向 Dev Cadence 维护者、流程评审者和希望了解交付模型的使用者。

目标仓库中的实际执行规则以 `src/workflows/refactor/SKILL.md` 生成的安装内容为准。本文用于解释业务流程设计，不替代 workflow skill。

## 目的

重构流程用于在不主动改变外部可观察行为的前提下，改善内部结构、边界、可维护性、可测试性或演进能力。流程必须先建立行为基线，再通过小步修改和回归验证证明既有外部契约得到保留。

## 适用场景

- 改善模块边界、依赖方向、代码结构、可读性、测试性或可维护性。
- 在保持公共 API、数据格式和外部集成契约兼容的前提下调整内部接口。
- 清理技术债、迁移内部实现或删除经过验证不再使用的旧路径。
- 实施一个以保持行为为前提的 `T-nnn` Technical Task。

## 不适用场景

- 新增能力或主动改变预期行为：使用 `feature-dev`。
- 修复已有行为偏差：使用 `bug-fix`。
- 创建、接收、登记或排序重构 Task：使用 `backlog`。

## 流程阶段

| 阶段 | 目标 | 主要产出 |
|---|---|---|
| 1. 需求确认 | 读取 Technical Task 卡片，明确结构问题、目标、影响范围、非目标、必须保持的行为、完成条件和风险；用户确认本次重构边界。 | 重构需求确认记录 |
| 2. 重构方案 | 分析当前结构与目标结构，选择重构方法，明确外部契约、行为基线、兼容策略、迁移方式、回滚点和风险。 | 重构方案 |
| 3. 实施计划 | 将重构拆成小步修改，安排行为基线补充、验证命令、调用点迁移、兼容层处理、旧路径删除和完成条件。 | 重构实施计划 |
| 4. 重构实施 | 按计划完成结构调整、测试资产和必要迁移；小步执行验证并完成代码审查。 | 可工作的交付物；测试资产；重构记录；代码审查报告 |
| 5. 回归验证 | 验证必须保持的行为没有回归，并确认结构目标、迁移目标和旧路径处理达到要求。 | 回归测试报告 |
| 6. 业务验收 | 用户根据结构目标、行为基线、回归结果和剩余风险决定接受、接受风险后接受或拒绝。 | 业务验收记录 |

## 工作项与行为基线

- 每次重构必须关联一个 `T-nnn` Technical Task 卡片。
- `refactor` 不创建或规范化 Task 卡片；缺失或不合规卡片必须先进入 `backlog`。
- Technical Task 卡片保存长期技术目标、原因、范围、约束、完成条件、状态和 Change Log。
- 需求确认记录保存本次运行使用的卡片版本、选定范围、明确排除项和用户确认。
- 行为基线必须绑定到重构前的版本和可重复证据，不能用重构后的行为重新定义原始预期。

## 关键规则

- 重构默认不改变外部可观察行为；任何主动行为变化都应切换为功能开发或拆成独立 Story。
- 进入重构实施前，重新核对卡片版本、已确认需求、重构方案、重构计划、权威设计资料、依赖和当前代码；只有受影响内容失效时才返回最早受影响阶段，无关变化不阻止重构。
- 公共 API、外部数据形状和集成契约必须保持兼容；允许收窄内部接口，但不能把公共契约变更伪装成重构。
- 重构方案必须定义行为基线，回归验证必须使用同一基线证明行为得到保留。
- 实施计划只服务于重构实施，不替代回归验证或业务验收。
- 开发实施包含结构调整及相应的单元测试、集成测试、契约测试、快照测试、检查脚本或检查清单。
- 渐进迁移、兼容层或多调用方场景必须明确调用方范围、迁移批次、兼容策略和旧路径删除门禁。
- 阻塞性验证结果不能进入业务验收；返工必须回到最早受影响阶段。
- 完成业务验收和 Completion 后，应将 Task 状态及交付引用回写到卡片和 Backlog。

## 与其他 Workflow 的关系

```text
backlog -> optional work-item-analysis -> T card -> refactor
```

用户可以直接提出重构请求，但 `using-dev-cadence` 必须先检查 Task 卡片及其 Backlog 行。缺失或不合规卡片先由 `backlog` 创建和确认，再进入 `refactor`。
