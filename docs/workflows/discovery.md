# 需求探索流程

本文说明 Dev Cadence 的 `discovery` 业务流程，包括适用场景、阶段目标、主要产出以及与其他 workflow 的边界。本文面向 Dev Cadence 维护者、流程评审者和希望了解需求治理模型的使用者。

目标仓库中的实际执行规则将由 `src/skills/discovery/SKILL.md` 定义并生成安装内容。本文用于说明业务流程设计，不替代 workflow skill。

## 目的

需求探索流程用于把不完整的想法、反馈、业务问题或产品方向整理成经过用户确认、可持续演进的 PRD。完整性是探索过程逐步形成的结果，不是输入前提。

## 适用场景

- 从一个模糊想法开始明确要解决的问题和产品方向。
- 建立新的 PRD。
- 在已有 PRD 上增量讨论新需求、范围变化、产品决策或未来方向。
- 澄清目标用户、业务价值、成功标准、范围、非范围、约束和开放问题。

## 不适用场景

- 将已确认 PRD 拆分成 Feature 和工作项：使用 `work-item-planning`。
- 对单个 Story、Bug 或 Technical Task 进行技术设计和开发交付：使用相应开发 workflow。
- 维护实现细节、代码变更或运行证据：由开发 workflow 负责。

## 流程阶段

| 阶段 | 目标 | 主要产出 |
|---|---|---|
| 1. 背景与问题探索 | 收集原始想法、反馈、业务问题和现状，明确动机、目标用户、问题陈述，并区分事实、假设和待确认内容。 | 探索上下文；问题陈述 |
| 2. 目标与价值定义 | 明确期望结果、业务价值、成功标准和不处理该问题的影响。 | 产品目标；用户与价值；成功标准 |
| 3. 范围与约束分析 | 明确当前范围、非范围、未来范围、业务规则、非功能要求、约束、风险、开放问题和候选决策。 | 范围边界；约束与风险；开放问题 |
| 4. PRD 建立或更新 | 将确认内容、草稿、开放问题、明确拒绝项、未来范围、产品级决策和 Change Log 写入 PRD。 | PRD 草案或增量变更 |
| 5. PRD 确认 | 向用户说明已确认内容、未解决问题、排除项和后续影响；由用户确认当前 PRD 版本能否作为规划基线。 | 已确认的 PRD 版本 |

## PRD 版本与历史

- PRD 使用从 `1` 开始的递增整数版本号。
- 业务目标、范围、成功标准、业务规则、约束或产品级决策发生实质变化时必须升版并增加 PRD 内的 Change Log。
- 仅修正拼写、排版或链接时不升版。
- Change Log 记录版本、日期、变化摘要和原因，不记录 Git commit hash。
- PRD 版本是需求基线的业务身份；Git 历史负责保存完整文件历史。
- 产品级 Decision 保存在 PRD 中，不要求目标仓库维护独立的 Decision Log。

## 关键规则

- 输入可以只有一个想法，不要求用户预先提供完整需求、验收标准或实现边界。
- `discovery` 同时支持首次建立 PRD 和增量更新已有 PRD，不单独增加 `update-prd` workflow。
- PRD 必须区分 `Confirmed`、`Draft`、`Open Question`、`Rejected` 和 `Future Scope`。
- 不得把草稿、建议或开放问题静默提升为已确认需求。
- 开放问题不必阻止整个 PRD 确认，只要已确认范围和未确认范围被明确隔离，且其影响得到说明。
- `discovery` 不创建 Feature、Story、Bug 或 Technical Task，不维护工作项 Roadmap。
- 后续规划或开发发现产品级需求缺失、冲突或变化时，应返回 `discovery` 更新并重新确认最早受影响的 PRD 版本。
- PRD 和其他需求资产保存在目标仓库正常项目空间，不得写入可替换的 `.dev-cadence/` 安装包。
- 不生成额外业务 Run Log；Dev Cadence manifest 和阶段记录保存 workflow 执行过程。

## 与其他 Workflow 的关系

```text
想法、反馈、业务问题或产品方向
              |
          discovery
              |
        已确认的 PRD
              |
    work-item-planning
```

用户只需要澄清或更新产品方向时，可以在 `discovery` 完成后停止；需要形成工作项组合时，再进入 `work-item-planning`。
