# Bug 修复流程

本文说明 Dev Cadence 的 `bug-fix` 业务流程，包括适用场景、阶段目标、主要产出以及与其他 workflow 的边界。本文面向 Dev Cadence 维护者、流程评审者和希望了解交付模型的使用者。

目标仓库中的实际执行规则以 `src/skills/bug-fix/SKILL.md` 生成的安装内容为准。本文用于解释业务流程设计，不替代 workflow skill。

## 目的

Bug 修复流程用于处理既有预期行为与实际行为之间的偏差。流程从问题报告开始，先验证问题性质和根因，再实施最小修复、执行影响范围回归验证并完成业务验收。

## 适用场景

- 用户报告错误、异常、失败测试、回归或其他非预期行为。
- 已实现能力没有满足已有需求、契约或行为基线。
- 实施一个已经规划的 `B-nnn` Bug。
- Technical Task 的实质目标是恢复错误行为，而不是新增能力或保持行为的结构改善。

## 不适用场景

- 用户希望主动改变既有预期行为：使用 `feature-dev`。
- 只改善内部结构且不改变预期行为：使用 `refactor`。
- 批量整理问题、分析优先级或维护 Bug Roadmap：使用 `work-item-planning`。

## 流程阶段

| 阶段 | 目标 | 主要产出 |
|---|---|---|
| 1. 问题诊断 | 读取 Bug 卡片，明确期望行为、实际行为、影响和已知条件；判断是否确认为 Bug；复现问题或记录替代证据；定位根因并确定修复边界。 | 问题诊断记录 |
| 2. 修复方案 | 基于诊断结论明确修复点、影响模块、兼容性要求、回归范围、风险和需要用户决策的取舍。 | 修复方案 |
| 3. 修复计划 | 将修复方案拆成可执行的 TDD 步骤，明确修复前失败证据、最小修改、配套测试、影响范围检查和完成条件。 | 修复计划 |
| 4. 修复实施 | 先建立能证明问题的失败测试或替代因果证据，再完成最小修复；运行开发阶段检查并完成代码审查。 | 可工作的交付物；测试资产；修复记录；代码审查报告 |
| 5. 回归验证 | 验证原问题不再出现，并检查修复影响范围内的相关行为没有回归；记录执行结果、跳过项、剩余风险和是否可以进入业务验收的结论。 | 回归测试报告 |
| 6. 业务验收 | 用户根据问题目标、修复结果、回归报告和剩余风险决定接受、接受风险后接受或拒绝。 | 业务验收记录 |

## Bug 卡片与诊断记录

- 每次 Bug 修复必须关联一个 `B-nnn` Bug 卡片。
- `bug-fix` 不创建 Bug 卡片；入口没有找到对应卡片时，必须先进入 `work-item-planning` 的单项登记模式。
- Bug 卡片保存长期问题身份，包括期望行为、已观察行为、影响、已知复现条件、状态、关系和 Change Log。
- Bug 卡片创建时不要求已经知道根因，也不要求所有复现条件已经明确。
- 问题诊断记录保存本次运行的复现证据、假设、根因、因果证据、修复边界和回归风险。
- 诊断完成后，只把必要摘要、状态、卡片版本和诊断记录引用回写到 Bug 卡片，不复制完整调查过程。

## 关键规则

- 问题诊断不只是整理用户描述；进入修复方案前必须确认问题性质、根因或足够的替代因果证据。
- 进入修复实施前，重新核对卡片版本、诊断、修复方案、修复计划、权威设计资料、依赖和当前代码；只有受影响内容失效时才返回最早受影响阶段，无关变化不阻止修复。
- 如果确认不是 Bug，应记录判断依据和用户确认，并以明确终态结束，而不是继续进入修复。
- 修复方案必须前置定义影响范围和回归范围，不能留到回归验证阶段临时决定。
- 修复计划只服务于修复实施，不替代回归验证或业务验收。
- 当存在自动测试或可重复检查时，应保留修复前失败和修复后通过的可追溯证据。
- 回归验证既要验证原问题已修复，也要验证修复影响范围内的相关功能没有回归。
- 阻塞性验证结果不能进入业务验收；返工必须回到最早受影响阶段。
- 完成业务验收和 Completion 后，应将 Bug 状态及交付引用回写到卡片和 Roadmap。

## 与其他 Workflow 的关系

```text
work-item-planning -> B card -> bug-fix
```

用户可以直接报告问题，但 `using-dev-cadence` 必须先检查 Bug 卡片。没有卡片时先由 `work-item-planning` 创建和确认卡片，再进入 `bug-fix`。Bug 卡片达到 `Ready` 只表示可以开始问题诊断，不要求已经确认根因。
