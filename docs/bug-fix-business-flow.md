# Bug 修复业务流程讨论总结

本文只总结本次讨论形成的结论，不依赖项目中已有的其他文档。当前只描述乐观路径，不展开角色分工、返工回环和异常处理。

| 步骤 | 做什么 | 产出物 |
|---|---|---|
| 1. 问题定位 | 用户报告问题；执行方明确问题现象、期望行为、实际行为和影响范围；判断是否确认为 bug；复现问题或说明无法复现；定位根因并给出证据。 | 问题定位记录 |
| 2. 制定修复方案 | 基于问题定位和根因，明确修复点、修复边界、会改哪些模块、可能影响哪些已有功能、需要回归验证的范围、风险和需要用户决策的取舍。 | 修复方案与影响范围 |
| 3. 制定修复计划 | 基于问题定位、修复方案和影响范围，制定开发实施阶段使用的 TDD 修复计划；明确复现 bug 的失败测试、修复步骤、配套测试、影响范围回归检查和完成条件。 | TDD 修复计划 |
| 4. 修复实施 | 按 TDD 修复计划先写能复现问题的失败测试，再完成最小化修复；运行开发阶段测试和检查；完成 code review；记录实际改动、测试、review 结论和剩余风险。 | 可工作的交付物；配套测试资产；修复记录 |
| 5. 回归验证 | 读取问题定位记录、修复方案与影响范围、TDD 修复计划、修复记录；验证原问题已修复；按影响范围验证相关功能没有回归；执行必要的构建、接口、页面或关键检查；记录测试用例、结果、跳过项和剩余风险；给出是否可以进入业务验收的结论。 | 回归测试报告 |
| 6. 业务验收 | 读取问题定位记录和验收标准；读取回归测试报告和剩余风险；试用、审阅或确认修复结果；判断修复是否满足业务目标；决定接受、拒绝，或接受剩余风险后接受；记录具名验收结论。 | 业务验收记录 |

## 关键约定

- `问题定位` 不只是整理用户描述，必须确认问题性质、复现情况、根因和证据。
- `制定修复方案` 必须前置考虑影响范围和回归范围，不能等到回归验证阶段才临时决定。
- `制定修复计划` 只服务于 `修复实施`，不负责制定业务验收计划。
- `修复实施` 包含实际修复及其配套的单元测试、集成测试、前端测试、mock、检查脚本或检查清单。
- `回归验证` 既要验证原问题不再出现，也要验证修复影响范围内的相关功能没有回归。
- `可工作的交付物` 可以是代码、配置、数据修正脚本、migration、文档、prompt、规则或其他项目资产。
