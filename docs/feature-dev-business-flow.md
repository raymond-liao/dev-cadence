# 功能开发业务流程讨论总结

本文只总结本次讨论形成的结论，不依赖项目中已有的其他文档。当前只描述乐观路径，不展开角色分工、返工回环和异常处理。

| 步骤 | 做什么 | 产出物 |
|---|---|---|
| 1. 需求确认 | 用户提出需求；执行方理解、整理并重述需求；做初步可行性判断；提出澄清问题、明显风险和需要后续技术方案处理的问题；用户确认修订后的需求版本。 | 明确的需求 |
| 2. 制定技术方案 | 基于明确的需求，判断实现路径是否清楚；必要时做技术方案调研；明确技术方向、系统边界、关键取舍、约束和风险；涉及业务取舍或风险接受时提交用户决策。 | 技术方案 |
| 3. 制定计划 | 基于明确的需求和技术方案，制定开发实施阶段使用的 TDD 开发实施计划；拆分开发实施模块和可验证行为场景；安排 TDD 小循环、mock、联调和接入顺序；明确每一步开发实施的产出物和完成条件。 | TDD 开发实施计划 |
| 4. 开发实施 | 按 TDD 开发实施计划完成代码、配置、文档、脚本、模板或其他项目资产；完成对应的单元测试、集成测试、前端测试、mock、检查脚本或检查清单；运行开发阶段测试和检查；记录实际完成内容。 | 可工作的交付物；配套测试资产；实施记录 |
| 5. 系统测试 | 读取明确的需求、技术方案、开发实施计划、实施记录；确认可工作的交付物、实际改动范围和运行入口；制定系统测试场景；执行系统测试；记录结果和剩余风险；给出是否可以进入业务验收的结论。 | 系统测试报告 |
| 6. 业务验收 | 读取明确的需求和验收标准；读取系统测试报告和剩余风险；试用、审阅或确认交付物；判断交付物是否满足业务目标；决定接受、拒绝，或接受剩余风险后接受；记录具名验收结论。 | 业务验收记录 |

## 关键约定

- `制定计划` 只服务于 `开发实施`，不负责制定系统测试计划或业务验收计划。
- `开发实施` 包含功能实现及其配套的单元测试、集成测试、前端测试、mock、检查脚本或检查清单。
- `系统测试` 关注整体系统是否可用，不替代开发实施阶段应完成的测试资产。
- `可工作的交付物` 可以是代码、配置、文档、脚本、模板、prompt 或其他项目资产。
