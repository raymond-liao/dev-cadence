# 重构业务流程讨论总结

本文只总结本次讨论形成的结论，不依赖项目中已有的其他文档。当前只描述乐观路径，不展开角色分工、返工回环和异常处理。

重构指在不主动改变外部可观察行为的前提下，改善内部结构、边界、可维护性、可测试性或演进能力。

| 步骤 | 做什么 | 产出物 |
|---|---|---|
| 1. 需求确认 | 用户提出重构目标；执行方明确要改善的结构问题、影响范围、非目标、必须保持的行为、验收标准和风险；用户确认修订后的重构需求。 | 明确的重构需求 |
| 2. 制定重构方案 | 基于明确的重构需求，分析当前结构问题和目标结构；选择重构方法；明确外部契约、行为基线、迁移策略、回滚点、风险和需要用户决策的取舍。 | 技术方案 |
| 3. 制定计划 | 基于明确的重构需求和技术方案，制定开发实施阶段使用的重构实施计划；拆分小步重构任务、行为基线补充、验证命令、调用点迁移、废弃路径删除和完成条件。 | 重构实施计划 |
| 4. 开发实施 | 按重构实施计划完成结构调整、配套测试资产和必要的迁移；小步执行验证；完成 code review；记录实际结构变化、测试、review 结论和剩余风险。 | 可工作的交付物；配套测试资产；重构记录；代码审查报告 |
| 5. 回归验证 | 读取重构需求、技术方案、行为基线、重构实施计划和重构记录；验证必须保持的行为没有回归；确认结构目标达成情况；记录测试用例、跳过项和剩余风险。 | 回归测试报告 |
| 6. 业务验收 | 读取重构需求、结构目标、回归测试报告和剩余风险；判断重构是否满足业务目标；决定接受、拒绝，或接受剩余风险后接受；记录具名验收结论。 | 业务验收记录 |

## 关键约定

- 重构默认不改变外部可观察行为；任何主动行为变化都应切换为功能开发，或拆成单独任务。
- 重构方案必须包含行为基线；回归验证必须回看行为基线。
- 制定计划只服务于开发实施，不负责制定业务验收计划。
- 开发实施包含结构调整及其配套的单元测试、集成测试、契约测试、快照测试、检查脚本或检查清单。
- 代码审查报告在开发实施阶段生成，是进入回归验证前的质量门。
- 可工作的交付物可以是代码、配置、脚本、测试资产、文档、prompt、规则或其他项目资产。
