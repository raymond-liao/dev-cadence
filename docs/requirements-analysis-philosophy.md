# Dev Cadence 的需求分析理念

## 需求分析的目标不是写文档

需求分析的目标，是帮助用户看清楚真正需要解决的问题，以及产品必须提供什么能力。

文档只是保存和传递产品结论的载体，不是产品需求本身，更不是需求分析的价值所在。一份格式完整、措辞专业的文档，仍然可能没有说明用户为什么需要产品、产品应该承担什么责任，也无法证明列出的功能能够解决真实问题。

Dev Cadence 不以“生成了一份需求文档”判断需求工作是否完成。它关注的是：产品理解是否已经形成，关键判断是否有依据，产品能力是否能够从真实用户和业务需求中推导出来。

## 需求是对产品责任的判断

用户最初提出的内容通常只是一个信号：一个想法、一项功能建议、一条反馈、一次失败、一个业务目标，或者一句“我们需要一个系统”。

这些输入很重要，但不能直接等同于需求。

```text
用户表达的方案 != 用户真正需要的结果
功能名称 != 产品能力
现象 != 根本问题
文档条目 != 已经理解的需求
```

需求分析需要把原始输入还原成对产品责任的判断：在明确的用户、场景和约束下，产品必须提供什么结果，才能帮助用户完成目标。

一个可靠的需求至少能够解释：

```text
谁
-> 在什么场景下
-> 想完成什么目标
-> 目前受到什么阻碍
-> 需要产品提供什么结果
-> 受到哪些业务规则和约束
-> 如何判断结果已经实现
```

只有这些问题被分析清楚，团队才有条件讨论产品能力、范围和交付计划。

## 从问题开始，而不是从功能开始

功能是解决问题的一种产品表达，不应该成为需求分析的起点。

当用户提出“增加审批功能”“做一个报表”“支持批量操作”时，Dev Cadence 不应立即把这些表述写成功能需求。它需要继续理解：

- 用户正在完成什么业务目标；
- 为什么当前方式无法满足目标；
- 哪些角色参与其中；
- 需要做出什么业务判断；
- 用户真正缺少的是信息、权限、协作、自动化，还是风险控制；
- 用户提出的功能是否是必要能力，还是未经验证的解决方案；
- 是否存在更小、更直接或更通用的产品能力。

需求分析不是否定用户提出的方案，而是避免产品准确实现一个错误假设。

## 从用户和业务中发现需求

产品能力必须来自用户和业务，而不是来自模板、代理补全或功能清单。

需求分析需要理解：

- 用户和业务角色；
- 各角色希望实现的目标；
- 目标发生的业务背景和触发条件；
- 用户从开始到获得结果所经历的完整过程；
- 过程中的决策、协作、信息和责任；
- 每个环节需要产生的业务结果；
- 正常、替代和失败情况下的不同需要；
- 法规、权限、时间、风险和一致性约束。

如果不知道用户是谁，就无法判断价值。如果不知道用户要完成什么，就无法判断功能是否必要。如果不知道业务过程和规则，就无法判断产品应该在哪个环节承担责任。

需求分析不能只观察用户明确说出的内容。它还需要发现流程断点、角色冲突、信息缺失、责任不清、未覆盖的异常情况，以及用户已经习惯但不应该继续承受的问题。

## 区分需要、约束和解决方案

需求讨论中经常同时出现三类不同内容：

- 需要：用户或业务必须获得的结果；
- 约束：产品必须遵守的业务、法规、风险或环境边界；
- 解决方案：实现结果的一种候选方式。

例如：

```text
需要：高风险操作必须经过授权后才能生效。
约束：授权过程必须可追溯，并满足职责分离要求。
解决方案：增加一个两级审批页面。
```

前两项定义产品责任，第三项只是候选机制。如果把解决方案直接写成需求，团队就会失去评估其他方案的空间，也可能在没有理解问题的情况下锁定错误实现。

Dev Cadence 必须忠实保存用户提出的方案，但不能因为用户提到了具体实现，就自动把它提升为已确认的产品需求。

## 从业务需求推导产品能力

产品能力不是功能名称的集合。它描述产品为了持续支持某类用户结果，必须具备什么稳定责任。

例如，“增加审批页面”是一个解决方案描述；“根据业务规则控制高风险操作的授权和责任确认”才更接近产品能力。

从业务需求推导产品能力时，需要判断：

- 多个用户问题是否由同一项能力解决；
- 一项能力是否混合了不相关的业务责任；
- 能力是否直接支持用户或业务结果；
- 能力边界是否明确；
- 能力需要哪些业务规则、信息和权限；
- 正常、替代和失败情况下是否都有合理结果；
- 能力是否可以独立验证其产品价值。

这个过程把零散输入整理成稳定的产品结构：

```text
用户目标
-> 业务过程
-> 业务需求
-> 产品能力
-> 可交付的产品行为
```

产品能力明确之后，团队才应该继续讨论具体交付内容和技术实现。

## 范围是对产品责任的边界判断

需求分析不仅决定产品要做什么，也决定产品不负责什么。

范围不能只是一张功能列表。它需要明确：

- 产品服务哪些用户和场景；
- 从哪里开始承担责任；
- 在哪里结束责任；
- 哪些结果由产品保证；
- 哪些结果依赖外部角色或系统；
- 哪些业务路径当前必须覆盖；
- 哪些内容被明确排除或延后；
- 哪些假设一旦不成立，就需要重新分析需求。

清晰的范围让产品能力保持聚焦，也让后续设计和开发能够识别越界变化。

## 完整结果比孤立功能更重要

用户需要的不是一组互不相干的功能，而是能够完成目标的产品。

需求分析需要从业务起点看到业务终点，确认产品能力组合起来能够产生完整结果。只实现若干高优先级功能，如果用户仍然无法完成一条端到端业务流程，就没有形成真实的产品价值。

最小产品范围也必须是完整的：它可以减少角色、场景、业务分支和自动化程度，但不能切断用户获得结果所必需的过程。

```text
业务起点
-> 必要的产品能力
-> 必要的用户行为
-> 业务终点
-> 用户获得可验证的结果
```

在最小完整结果成立后，产品再逐步覆盖更多合法选择、异常情况、边界场景和服务能力。

## 异常和替代情况属于需求本身

真实业务不会永远按照理想条件运行。

用户可能改变选择，业务规则可能拒绝操作，外部参与方可能没有响应，数据可能不完整，权限可能不足，流程也可能需要恢复或终止。

这些情况不是开发或测试阶段才需要补充的细节。它们决定产品在真实环境中是否可信，因此属于需求分析的一部分。

需求分析需要明确：

- 正常情况下如何获得结果；
- 存在其他合法选择时如何继续；
- 条件不满足时产品如何拒绝；
- 失败发生后是否可以恢复；
- 无法继续时如何安全结束；
- 用户如何理解当前结果和下一步动作。

## 不确定性必须被看见

需求分析不可能在任何时候都拥有完整信息。可靠的做法不是把空白补成看似确定的内容，而是区分不同可信程度的判断。

Dev Cadence 需要明确区分：

- 已确认的事实和需求；
- 有证据支持但尚未确认的假设；
- 用户或代理提出的候选方案；
- 存在冲突的来源；
- 仍需回答的开放问题；
- 已明确拒绝或延后的方向。

不确定性本身不是失败。隐藏不确定性、把建议写成结论，才会让后续产品和技术决策建立在错误基础上。

## 证据决定结论的可信度

需求结论应尽量建立在可以检查的依据上，例如：

- 用户反馈和实际行为；
- 业务流程和操作记录；
- 已确认的业务规则；
- 法规、合同和组织约束；
- 当前产品行为和已知缺口；
- 可测量的业务结果；
- 用户明确作出的选择和拒绝。

当多个来源冲突时，不能为了形成整齐结论而静默选择其中一个。需要说明冲突是什么、会影响哪些产品判断，以及由谁在什么阶段解决。

## 可追溯性服务于变化，而不是审计形式

产品需求一定会变化。可追溯性的价值，是帮助团队在变化发生时理解影响，而不是为了留下更多记录。

Dev Cadence 希望保持这样一条产品决策链：

```text
业务问题
-> 用户和目标
-> 业务过程
-> 业务需求
-> 产品能力
-> 可交付的产品行为
-> 开发和验证结果
```

当其中任何一层变化时，团队可以沿着关系判断：

- 哪些产品结论已经失效；
- 哪些业务流程或规则受到影响；
- 哪些产品能力需要调整；
- 哪些正在进行的交付需要重新确认；
- 哪些验证证据需要重新获得。

可追溯性最终服务于正确决策和安全变化，而不是服务于文档数量。

## 文档只是产品理解的持久化载体

需求分析会产生文档，但文档的作用是保存、传递和验证已经形成的产品理解。

文档应该帮助团队：

- 复用已经确认的事实和判断；
- 理解需求来自哪里；
- 区分正文结论与未决问题；
- 在跨会话和跨团队协作时保持上下文；
- 判断变化影响；
- 为后续规划、设计、开发和验证提供稳定依据。

如果产品理解发生变化，相关文档必须协调更新。如果文档内容无法追溯到用户、业务目标和已确认判断，即使格式正确，也不能成为可靠的产品依据。

文档数量不是成熟度，模板完成度不是需求质量，语言流畅也不能证明产品判断正确。

## AI 应该帮助分析，而不是代替理解

AI 很容易快速生成结构完整、措辞专业的需求文档，但流畅表达不能证明需求正确。

在 Dev Cadence 中，AI 的职责是：

- 帮助识别原始输入中的问题、目标、假设和解决方案偏见；
- 发现缺失的用户、场景、业务环节和异常情况；
- 组织不同来源的信息并指出冲突；
- 提出可比较的产品方向和能力边界；
- 保持需求、产品能力和交付工作之间的关系；
- 在证据不足时保留开放问题，而不是自动补全；
- 将已经形成的产品理解持久化为清晰、可维护的资产。

AI 不应该为了完成流程而虚构用户、需求、业务规则或成功标准，也不应该把自己的建议伪装成用户已经确认的产品结论。

## 怎样判断需求分析已经产生价值

需求分析是否有效，不由文档是否完成决定，而由团队能否清楚回答以下问题决定：

- 我们正在为谁解决什么问题；
- 这个问题为什么值得解决；
- 用户需要完成什么业务目标；
- 产品应该在哪些业务环节承担责任；
- 产品必须提供哪些能力；
- 产品明确不负责什么；
- 正常、替代和失败情况下分别需要什么结果；
- 什么是最小但完整的端到端价值；
- 如何判断产品已经成功解决问题；
- 哪些结论已经确认，哪些仍然是假设或开放问题；
- 当需求变化时，哪些产品和交付内容会受到影响。

如果这些问题仍然无法回答，继续扩充文档不会使需求变得更清楚。团队需要回到用户、业务目标和真实场景继续分析。

## Dev Cadence 的需求分析承诺

Dev Cadence 不以生成更多文档为目标。它希望帮助用户完成一系列可靠的产品判断：

```text
我们理解了什么问题
-> 我们确认了哪些用户和目标
-> 我们看清了怎样的业务过程
-> 我们发现了哪些真实需求
-> 我们决定产品承担什么责任
-> 我们定义产品必须具备什么能力
-> 我们找到了最小的完整价值
-> 我们知道接下来为什么交付这些工作
```

文档只是这些判断的持久化结果。

Dev Cadence 真正要保护的，是从用户和业务出发形成产品能力的过程，以及这份产品理解在后续规划、设计、开发、验证和演进中不被丢失。
