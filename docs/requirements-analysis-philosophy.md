# Dev Cadence 的单卡需求分析理念

## 目标是让一张卡足以进入实施

Dev Cadence 的需求分析不负责发现产品方向，也不负责建立 PRD、User Journey、Feature 或 Story Map。它只判断一张 Story、Task 或 Bug 是否已经清楚到足以进入对应的软件实施流程。

一张卡片至少需要让实施者理解：

```text
为什么要做
-> 本次要产生或恢复什么结果
-> 哪些内容属于范围
-> 哪些内容明确不属于范围
-> 如何判断工作完成
-> 哪些约束、依赖或问题会阻止实施
```

## 卡片准入与需求分析是不同责任

`backlog` 负责创建卡片、校验用户提供的卡片、登记 Backlog 和维护实施顺序。`work-item-analysis` 只负责一张已有卡片的详细定义。

符合 Dev Cadence 卡片和成熟度契约的外部卡片可以直接进入 Backlog，不需要重新建卡或重复分析。不符合要求的外部卡片保持原状，并作为新需求输入走标准建卡流程。

格式合规不等于需求成熟。卡片字段齐全但目标、范围或完成条件不明确时，仍需要单卡分析；反之，已经完整且成熟的卡片不应为了流程形式再次分析。

## 每次只分析一张卡

单卡边界让目标、范围、Version、Status、Change Log 和用户确认保持一一对应。跨卡组合规划、批量需求分析和整个 Backlog 的产品结构推导不属于 Dev Cadence。

相关卡片可以作为证据读取，但不能把它们静默并入当前分析。发现重叠、冲突或新的独立目标时，应把问题暴露给用户，而不是在一次分析中自动合并或扩展范围。

## Story、Task 与 Bug 的成熟度不同

Story 需要明确角色、目标、价值、范围、可观察行为、验收条件、依赖和阻塞性 Open Questions。只有这些内容明确且经过用户确认，Story 才能达到 `Ready` 并进入 `feature-dev`。

Task 需要明确目标、必要性、范围、完成条件、影响区域、约束和依赖。Task 不强制使用 Story 的 `Ready` 门，但对应 Delivery Workflow 在改代码前仍须确认目标、范围和完成条件。

Bug 需要明确预期行为、观察行为、影响和现有失败证据。Bug 不要求预先知道技术根因；复现、因果分析、修复边界和回归证明属于 `bug-fix`。

## 不确定性必须保持可见

分析必须区分已确认事实、假设、用户提出的候选方向、冲突来源和 Open Questions。缺少关键信息时，正确动作是让卡片保持未满足实施条件，而不是用代理推断补成确定结论。

不确定性只有在会改变当前卡片的目标、范围、完成条件或实施安全性时才阻塞。与当前交付无关的问题不应被提升为额外产品分析工作。

## 分析结果仍然属于卡片

卡片是长期权威的工作项定义。`work-item-analysis` 不创建 Delivery manifest、技术方案、实施计划、测试报告或业务验收记录。

确认的实质变化更新卡片 Version 和 Change Log；纯格式、链接或执行状态变化不提升 Version。卡片 Status 或 Version 变化时，只机械同步 Backlog 对应行，不改变待处理顺序。
