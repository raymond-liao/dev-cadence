## 当前结论

在平台支持内部子代理时，`using-dev-cadence` 必须把完整的 Dev Cadence 任务交给一个主执行子代理。主会话只保留用户决定、需要用户输入的阻塞和最终结论；主执行子代理负责入口路由、workflow 恢复、调查、草稿、文件修改、实现、测试、Review、验证、阶段记录和受既有授权门禁约束的 Git 操作。

主执行子代理与普通子任务代理必须有明确角色边界：前者可选择或恢复 workflow，并可继续委派边界清楚的子任务；后者不得重复路由或递归委派完整任务。主执行子代理不可恢复时，另一个主执行子代理必须能依据文件证据恢复。

## ✅ 范围

- 在平台支持内部子代理时，从 `using-dev-cadence` 入口委派完整 Dev Cadence 请求。
- 定义主会话、主执行子代理和普通子任务代理的责任与禁止递归行为。
- 让已安装 workflow 的全部非交互工作在主执行子代理侧连续完成。
- 规定只在需要用户决定、用户输入阻塞或任务终态时回到主会话，并限制返回内容为结论、选项、影响、风险和证据路径。
- 允许未确认草稿使用临时或缓存位置，但明确其非权威性质，并在确认前保持权威资产不变。
- 保持 merge、PR、push、discard 和分支删除的既有用户授权门禁。
- 支持原主执行子代理回收用户回应，或在其不可恢复时从文件证据续跑。
- 平台不支持内部子代理时保留现有主会话执行方式。
- 同步 source、dist、安装包、根目录 `version` 和契约测试。

## ❌ 非范围

- 修改根 `AGENTS.md` 或 `src/AGENTS-snippet.md`。
- 新增 workflow、Skill、脚本、配置项或公开状态。
- 修改 vendored Superpowers，或在本 Story 中实现平台本身缺失的子代理能力。
- 取消用户确认门、转移业务决定所有权，或扩大 Git 操作授权。
- 将未确认草稿提升为权威资产或长期业务来源。
- 承诺系统临时或缓存文件跨机器、系统清理或执行环境销毁后的可恢复性。

## 验收标准

1. 主会话只读取入口规则后即可委派完整任务，无需加载具体 workflow 或探索仓库。
2. 主执行子代理可选择或恢复 workflow；普通子任务代理不会重复路由或递归派发完整任务。
3. 所有已安装 workflow 的非交互工作均可在子代理侧连续执行。
4. 主执行子代理只在需要用户决定、必须由用户提供信息的阻塞或终态返回主会话。
5. Asset Workflow 可使用临时非权威草稿，且用户确认前不修改权威资产。
6. 主会话确认信息包含当前结论、完整用户选项、选项影响、风险和证据路径，不包含不必要的过程输出。
7. Git 操作由主执行子代理执行，但既有 merge、PR、push、discard 和分支删除授权边界不变。
8. 用户回应可交回原主执行子代理，或由新的主执行子代理从文件证据恢复。
9. 无内部子代理能力的平台继续使用现有主会话执行方式。
10. 契约测试覆盖角色边界、返回条件、临时草稿边界、Git 授权、fallback 以及 source、dist 和安装结果同步。

## 风险与待确认假设

- 平台如何识别“内部子代理可用”需要在 Technical Solution 中映射到现有平台语义，且不得通过新增公开配置规避该边界。
- 委派只能改变执行位置，不能把任何用户确认、Business Acceptance 或 Completion 决定自动化。
- 当前阶段未发现新的业务需求；技术实现选择尚未确认。
