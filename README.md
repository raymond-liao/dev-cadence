# Dev Cadence

Dev Cadence is a delivery-governance package for AI coding agents. It keeps implementation work reviewable and recoverable from work-item intake through design, planning, implementation, verification, acceptance, and Git integration.

Dev Cadence does not perform product discovery, PRD creation, User Journey analysis, Feature definition, Story Map, Milestone, MVP, or portfolio planning. Product analysis can happen elsewhere; implementation work enters Dev Cadence through its card and Backlog contract.

## Quickstart

Install Dev Cadence into another repository:

```bash
git clone <your-dev-cadence-repository-url>
cd dev-cadence
bash scripts/install.sh /path/to/target-repository
```

Then copy the Markdown snippet from:

```text
/path/to/target-repository/.dev-cadence/AGENTS-snippet.md
```

into the target repository's root `AGENTS.md`.

## Delivery Model

Dev Cadence owns implementation work items and a repository-local Backlog:

```text
new request or existing card
-> backlog admission
-> single-card analysis when needed
-> docs/backlog.md
-> claim
-> feature-dev | bug-fix | refactor
-> solution, implementation plan, implementation, review, verification, acceptance
-> card and Backlog lifecycle writeback
```

Cards use stable repository-global IDs:

- Story: `S-nnn`
- Task: `T-nnn`
- Bug: `B-nnn`

An existing card that fully satisfies the Dev Cadence card and maturity contract can be registered in Backlog without recreation or repeated analysis. A nonconforming supplied card is preserved and treated as input to the standard new-request path.

## Workflows

### backlog

Creates Story, Task, and Bug cards; validates supplied cards; registers conforming cards; and owns `docs/backlog.md` structure, lifecycle sections, and recommended pending order.

Backlog does not perform product analysis, detailed single-card analysis, diagnosis, implementation, testing, or acceptance.

### work-item-analysis

Analyzes exactly one existing Story, Task, or Bug card. It clarifies the goal, scope, expected behavior or completion conditions, dependencies, constraints, and Open Questions.

Story `Ready` is decided from the Story's own implementation definition. It does not require a PRD, Feature, User Journey, or Story Map reference. Bug root-cause analysis remains in `bug-fix`.

### architecture-design

Creates or updates one standalone architecture asset for an explicit architecture goal. It does not replace a Delivery Workflow's task-scoped solution.

### feature-dev

Delivers new or intentionally changed user-visible or system-visible behavior from an admitted Story or Task.

```text
Requirements Confirmation -> Technical Solution -> Implementation Plan -> Implementation -> System Testing -> Business Acceptance -> Completion
```

### bug-fix

Diagnoses and repairs broken existing expected behavior from an admitted Bug card.

```text
Problem Diagnosis -> Repair Solution -> Repair Plan -> Repair Implementation -> Repair Verification -> Business Acceptance -> Completion
```

### refactor

Improves internal structure without intentionally changing expected behavior from an admitted Task card.

```text
Refactor Scope -> Refactor Solution -> Refactor Plan -> Refactor Implementation -> Refactor Verification -> Business Acceptance -> Completion
```

## Assets And Records

Asset Workflows maintain long-lived authoritative files under `docs/`:

```text
docs/backlog.md
docs/stories/S-nnn-<slug>.md
docs/tasks/T-nnn-<slug>.md
docs/bugs/B-nnn-<slug>.md
docs/architecture/<goal-slug>.md
```

Delivery Workflows maintain recoverable evidence under:

```text
build/dev-cadence/<workflow>/<task-slug>/
```

Delivery records include the active manifest, confirmed inputs, task-scoped solution, implementation plan, implementation evidence, code review, system verification, business acceptance, Git integration, and cleanup evidence. They are not stored inside the installed `.dev-cadence` package.

## Configuration

Copy the example configuration to the target repository root:

```bash
cp .dev-cadence/.dev-cadence.example.yaml .dev-cadence.yaml
```

Supported fields:

```yaml
output_language: en

worktree:
  enabled: false
  directory: .worktrees
```

`.dev-cadence.yaml` is target-repository runtime configuration and should not be committed unless that repository explicitly chooses to version it.

## Runtime Rules

- `.dev-cadence/` is replaceable installed package content.
- Work-item cards, Backlog, architecture assets, and delivery records belong to the target repository, not the package.
- Installing or upgrading Dev Cadence never authorizes deletion of existing product documents or other user data.
- `src/workflows/**` is the authoritative source for workflow behavior in this repository; `dist/.dev-cadence/**` is generated by `bash scripts/build.sh`.
