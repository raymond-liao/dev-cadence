## Workflow

### Stage 1: Build Report Outline

Use the structure in `references/methodology/output-template.md`.

Ensure the report answers:

- What does the current proposal get right?
- What needs clarification or revision?
- What should be in the MVP?
- What should be in the team Skill?
- What should wait for platformization?
- What risks and evaluation criteria matter most?

### Stage 2: Incorporate Verification Constraints

Use the Argument Verification Report to:

- Remove unsupported claims.
- Soften assumptions.
- Mark external research gaps if external research was not executed.
- Keep recommendations traceable.

### Stage 3: Write Final Report

Write to:

```text
{PROJECT_ROOT}/workspace/04.reports/final_report.md
```

### Stage 4: Optional Output Copy

If useful, also copy or summarize the final report under:

```text
{PROJECT_ROOT}/_output/
```

Only do this after the main report exists.

## Output Requirements

Follow `references/methodology/output-template.md`.

The final report must include:

- Executive summary.
- Current proposal map.
- Core judgment validation.
- Software delivery alignment analysis.
- Multi-Agent architecture evaluation.
- Team Skill delivery design.
- MVP scope and validation plan.
- Risk matrix and quality gate recommendations.
- README revision recommendations.
- Next-step actions.
- References.
